# AGENTS.md

## 프로젝트 원칙
- 이 프로젝트는 안정성을 최우선으로 한다. 기능 추가나 속도 개선보다 버그 가능성 감소를 우선한다.
- 이전 v1 프로젝트의 코드, 구조, DB 스키마, 구현 패턴을 재사용하지 않는다.
- 전략, 주문, 데이터, 리스크, UI, 로그, 설정은 모듈 책임을 분리한다.
- 브로커 계층은 `BrokerInterface` 추상화 뒤에 KIS 구현체를 둔다.
- 모의투자와 실계좌는 코드, 설정, UI에서 명확히 분리한다. 실계좌 주문에는 별도 확인 게이트를 둔다.
- 모든 주문은 중복 방지 장치를 둔다. 주문 키는 메모리 가드와 DB 영속화 둘 다 사용한다.
- 모든 상태는 영속화한다. 재시작 후에도 보유 포지션, 주문 상태, 핵심 이벤트를 복구할 수 있어야 한다.
- 로그인 실패, API 지연, 데이터 누락, 미체결, 주문 실패는 예외가 아니라 기본 운영 시나리오로 설계한다.
- 하드코딩을 금지한다. 파라미터는 설정 파일 또는 UI에서 조정 가능해야 한다.
- 큰 기능은 단계적으로 개발한다. 각 단계는 독립적으로 동작하고 검증 가능해야 한다.

## 응답 방식
- 코드 작성 전 반드시 아래 순서로 정리한다.
  1. 요구사항 요약
  2. 가정 및 제약사항
  3. 설계 개요
  4. 위험 요소
  5. 코드 또는 변경 사항
  6. 테스트 방법
- 부분 수정이라도 전체 영향 범위를 함께 설명한다.
- 비개발자도 이해할 수 있게 쉽고 짧게 설명한다.
- 문제 원인 또는 구현 의도, 핵심 변경점, 부작용 가능성, 추가 점검 포인트를 포함한다.
- 기존 기능이 깨질 수 있는 회귀 위험을 따로 표시한다.

## 자동매매 안전 기준
- 자동매매 코드는 예외처리, 로그, 중복 방지, 상태 영속화를 기본으로 포함한다.
- 주문, 체결, 조회, 전략 판단 로직은 분리한다.
- 실시간 처리에는 중복 실행 방지 장치를 둔다.
- 전략 조건과 실행 조건은 분리한다.
- 로그인 실패, API 응답 지연, 데이터 누락, 장중 미체결, 중복 주문, 잘못된 재시도를 반드시 고려한다.
- `try/except: pass` 같은 예외 무시는 금지한다.
